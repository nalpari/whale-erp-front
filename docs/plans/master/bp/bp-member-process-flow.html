Overview 전체 프로세스 개요

BP 등록부터 최초 로그인까지의 전체 흐름을 2단계로 나누어 설명합니다.

STEP 1. BP 등록

관리자가 Business Partner를 등록하면
로그인용 Member 계정이 자동 생성됩니다.

  • Organization(BP) 생성
  • Member + MemberDetail 생성
  • masterId → loginId
  • 초기 비밀번호 = masterId
  • 관리자가 BP 사용자에게 ID 전달

STEP 2. 최초 로그인

BP 사용자가 최초 로그인하면
비밀번호 변경 모달이 강제 표시됩니다.

  • loginId와 동일한 비밀번호로 로그인
  • 비밀번호 변경 모달 (닫기 불가)
  • 변경 완료 후 시스템 사용 가능
  • 변경 전까지 어떤 기능도 사용 불가
초기 비밀번호 정책: 초기 비밀번호는 Master ID와 동일하게 설정됩니다. 별도의 이메일 발송이나 랜덤 비밀번호 생성 없이, 관리자가 BP 사용자에게 직접 Master ID를 전달하면 됩니다. 보안을 위해 최초 로그인 시 반드시 비밀번호를 변경해야 합니다.
참고: 권한(Authority) 부여는 별도 작업으로 진행됩니다. Member 생성 시점에는 권한이 없으므로, 관리자가 사후에 권한을 부여한 뒤 로그인이 정상 동작합니다.

STEP 1 BP 등록 프로세스 상세

관리자가 BP를 등록할 때의 시스템 내부 처리 흐름입니다.

1. 관리자가 BP 등록 폼 작성
업체명, 사업자등록번호, 대표자 정보 등 기본 정보 입력
Master ID = 로그인에 사용될 ID (영문+숫자 8자 이상)
2. Master ID 중복 확인
입력한 Master ID가 이미 사용 중인지 검증
GET /api/auth/check-login-id?loginId={masterId}
3. 저장 버튼 클릭 → API 호출
폼 데이터 + LNB 로고 파일을 multipart/form-data로 전송
POST /api/master/bp
4. Organization(BP) 생성
공통코드 검증 → 사업자번호 중복 검증 → Organization 엔티티 저장 → BP코드 자동 생성 (PTN000001)
5. Member 계정 자동 생성 NEW
Organization 저장 후 Member + MemberDetail을 함께 생성
초기 비밀번호 = masterId (로그인 ID와 동일)
Member(loginId=masterId, passwordHash=encode(masterId), passwordChangeRequired=true)

Member 생성 상세

항목값
loginIdmasterId (폼에서 입력)
name대표자명
passwordHashmasterId를 BCrypt 해시 (초기 비밀번호 = loginId)
email대표자 이메일
isOperate1 (활성)
passwordChangeRequiredtrue 강제 변경

MemberDetail 생성 상세

항목값
member위에서 생성한 Member
organization새로 생성된 Organization
mobilePhone대표자 휴대폰번호
초기 비밀번호 정책:
초기 비밀번호는 Master ID와 동일하게 설정됩니다.
passwordEncoder.encode(masterId)
예시: masterId가 partner001이면 비밀번호도 partner001

STEP 2 최초 로그인 & 비밀번호 변경 강제

BP에서 생성된 계정으로 최초 로그인 시의 흐름입니다. 비밀번호 변경 전까지 시스템을 사용할 수 없습니다.

1. 로그인 페이지에서 인증
관리자에게 전달받은 Master ID(= loginId)와 동일한 초기 비밀번호로 로그인
POST /api/auth/login
2. 서버 인증 처리
loginId 조회 → 비밀번호 BCrypt 검증 → JWT 발급
응답에 passwordChangeRequired: true 포함
3. 프론트엔드에서 응답 확인
passwordChangeRequired === true 이면
auth-store에 상태 저장 → 비밀번호 변경 모달 호출
4. 비밀번호 변경 모달 (강제)
닫기 불가 (ESC, 백드롭 클릭, X 버튼 없음)
현재 비밀번호 + 새 비밀번호 + 확인 입력
PUT /api/auth/change-password
5. 변경 완료 → 시스템 사용 가능
passwordChangeRequired → false 업데이트
모달 닫힘 → 정상 메인 화면 진입

API 로그인 응답 변경 & 비밀번호 변경 API

기존 로그인 API 응답에 passwordChangeRequired 필드가 추가되고, 비밀번호 변경 전용 API가 신규 생성됩니다.

로그인 응답 (변경 후)

POST /api/auth/login
Response 200 OK

{
  "accessToken": "eyJhbG...",
  "refreshToken": "eyJhbG...",
  "tokenType": "Bearer",
  "expiresIn": 3600,
  "loginId": "masterid001",
  "name": "홍길동",
  "mobilePhone": "01012345678",
  "passwordChangeRequired": true,
  "authority": null,
  "companies": [...]
}
passwordChangeRequired 값에 따른 분기:
• true → 비밀번호 변경 모달 강제 표시
• false → 기존 흐름 (권한 선택 또는 메인 진입)

비밀번호 변경 API NEW

PUT /api/auth/change-password
Authorization: Bearer {accessToken}

Request
{
  "currentPassword": "masterid001",
  "newPassword": "NewPass1234!"
}

Response 200 OK
{
  "code": "SUCCESS",
  "message": "비밀번호가 변경되었습니다."
}

서버 처리 흐름

#처리 내용
1JWT에서 loginId 추출
2Member 조회
3currentPassword BCrypt 검증
4newPassword 형식 검증 (영문+숫자+특수 8~20자)
5currentPassword ≠ newPassword 확인
6passwordHash 업데이트
7passwordChangeRequired = false
